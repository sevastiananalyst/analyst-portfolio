# Нефункциональные требования к системе документооборота (Битрикс24)

## Проект: строительство метро «Юго-Западная», ООО «Розенберг»

### 1. Производительность

1.1. Время открытия карточки документа не должно превышать 2 секунд при штатной нагрузке (до 50 одновременных пользователей).

1.2. Загрузка файла размером 20 МБ должна занимать не более 10 секунд (при скорости интернета от 10 Мбит/с).

1.3. API интеграции с генподрядчиком должен отвечать не более 3 секунд, иначе запрос повторяется с экспоненциальной задержкой (до 3 раз).

### 2. Надёжность

2.1. Доступность системы в рабочие часы (пн–пт, 9:00–19:00) — 99.5%.

2.2. Плановые обновления проводятся по субботам с 22:00 до 2:00, с уведомлением за 3 дня.

2.3. Автоматическое резервное копирование всех документов и метаданных в PostgreSQL — ежедневно, хранение 30 дней.

### 3. Безопасность

3.1. Ролевая модель:
- Прораб: создание/редактирование своих документов, просмотр статусов
- ПТО: проверка документов всех объектов, запуск согласования
- Руководитель: просмотр дашборда, отчётов, эскалация
- Администратор: полный доступ, настройка БП

3.2. Все действия пользователей логируются в Confluence или отдельной таблице БД.

3.3. Гостевой доступ для генподрядчика — только по одноразовой ссылке, срок жизни 7 дней, без права скачивать другие документы.

### 4. Удобство использования (Usability)

4.1. Интерфейс должен быть интуитивно понятным для пользователей с базовыми навыками ПК (возрастная группа 40+ лет, прорабы и мастера).

4.2. Кнопки основных действий («Отправить на проверку», «Утвердить», «Отклонить») должны быть крупными и контрастными.

4.3. Обязательные поля при загрузке документа подсвечиваются звёздочкой, при пропуске — выдаётся понятное сообщение об ошибке.

### 5. Масштабируемость

5.1. Система должна поддерживать до 5000 документов в год без деградации производительности.

5.2. При добавлении новых объектов строительства (новые станции метро) должна быть возможность расширения справочников без перепрограммирования.
