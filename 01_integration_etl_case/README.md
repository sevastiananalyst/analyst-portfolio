# Кейс 1: Интеграции и ETL для геоинформационной системы (ГИС)

## Контекст
Компания **Гортис АТИ** (системная интеграция, автоматизация предприятий). Проект: разработка и внедрение геоинформационной системы для заказчика из сферы транспорта и инфраструктуры.

Задачи:
- Загрузка пространственных объектов (дороги, здания, инженерные сети) из внешних источников (Excel, CSV, шейп-файлы) в PostgreSQL/PostGIS.
- Настройка интеграций между ГИС и смежными системами через REST API и очереди (Kafka, RabbitMQ).
- Очистка, валидация и дедупликация данных.
- Проектирование API-контрактов для обмена данными.

## Моя роль
Системный аналитик:
- Сбор и анализ требований к данным и интеграциям.
- Формализация ТЗ, проектирование API (OpenAPI).
- Разработка ETL-процессов на Python (загрузка из Excel/CSV в PostgreSQL).
- Написание сложных SQL-запросов для валидации и дедупликации.
- Настройка асинхронного обмена через Kafka (события об изменении объектов).

## Результаты
- Загружено **1000+ объектов** в ГИС, время подготовки данных сокращено на **35%**.
- Устранены ключевые противоречия в исходной информации → снижение количества инцидентов на **50%**.
- Разработаны API-контракты, которые используются в продуктивной среде.

## Артефакты в этой папке

### API и веб-хуки
- `api_specs/openapi.yaml` — спецификация REST API для приёма данных о новых объектах.

### ETL и SQL
- `python_etl/etl_load_leads.py` — скрипт загрузки данных из Excel в PostgreSQL с очисткой и дедупликацией.
- `sql_etl/lead_deduplication_window.sql` — SQL-запрос с оконной функцией ROW_NUMBER() для удаления дублей.

### Моделирование данных
- `models/er_diagram.puml` — ER-диаграмма базы данных ГИС (объекты, атрибуты, связи).

### Асинхронные очереди (Kafka)
- `kafka_example/producer.py` — пример отправки события о создании нового объекта.
- `kafka_example/consumer.py` — пример получения события и обновления статуса.

Все артефакты основаны на реальных задачах проекта, но обезличены для соблюдения NDA.
