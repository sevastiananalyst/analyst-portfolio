# Кейс 1: Интеграции, ETL и работа с данными

## Контекст
В рамках проекта по автоматизации документооборота (строительство метро "Юго-Западная") потребовалось:
- Настроить обмен данными между Битрикс24 и внешними системами (генподрядчик, Росстройконтроль)
- Разработать ETL-процессы для загрузки справочников и актов из Excel/CSV в PostgreSQL
- Реализовать дедупликацию и очистку данных
- Обеспечить асинхронное взаимодействие через очереди (Kafka, RabbitMQ)

## Моя роль
Системный аналитик: проектирование API-контрактов, написание ETL-скриптов на Python, оптимизация SQL-запросов, настройка очередей.

## Что сделано

### API и веб-хуки
- Спроектирован REST API для приёма актов от генподрядчика (см. `api_specs/openapi.yaml`)
- Настроены веб-хуки в Битрикс24 для отправки уведомлений в Telegram при изменении статуса документа

### ETL и SQL
- Написан Python-скрипт для выгрузки актов из Excel в PostgreSQL с очисткой телефонов и дедупликацией (`python_etl/etl_load_leads.py`)
- Создан сложный SQL-запрос с оконной функцией для выявления дублей документов (`sql_etl/lead_deduplication_window.sql`)

### Асинхронные очереди (Kafka / RabbitMQ)
- Спроектирована схема событий: `lead_created`, `lead_duplicate`, `doc_approved`
- Пример конфигурации producer/consumer (лежит в `kafka_example/` — см. файлы)

## Результаты
- Сокращение времени подготовки данных на **35%** (автоматическая загрузка 1000+ объектов)
- Снижение количества ошибок в данных на **50%** (благодаря валидации и дедупликации)
- Полная прозрачность обмена документами с генподрядчиком и Росстройконтролем

## Артефакты в этой папке
- `api_specs/openapi.yaml` — спецификация REST API
- `sql_etl/lead_deduplication_window.sql` — SQL с оконными функциями
- `python_etl/etl_load_leads.py` — ETL-скрипт на Python
- `kafka_example/` — примеры producer/consumer на Python
- `models/` — ER-диаграммы и схемы данных (PlantUML)
